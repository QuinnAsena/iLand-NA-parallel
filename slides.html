Automating iLand on HPC

From hard-coded scripts to parallel, replicated scenario runs

Parallel iLand runs on HPC

When this is necessary:

  • running replicates in parallel with a JavaScript dependency (e.g., saving fire outputs).
  • running on an HPC system where iLand’s internal output structure needs redirecting.

Why this is necessary:

  • efficient use of HPC resources.
  • With our 60 kha landscape, individual runs take 4–8 hours at fine sapling resolution, but only use around a third of a node.
  • A single job on Derecho books the whole node’s resources.

The original scripts were legacy scripts with hard-coded pathways not intended for HPC use in parallel.

Note

Not necessary if you do not have a JavaScript dependency or if you can run replicates sequentially.

Defensive XML defaults

I set placeholder values in the project file tags so iLand errors loudly rather than silently falling back to leftover inputs:

<database>
  <in>species_params_overwritten_by_csv</in>
  <out>out_overwritten_by_csv</out>
  <climate>gcm_overwritten_by_csv</climate>
</database>

Tip

If a CSV value is missing or mis-spelled, iLand will fail to find the file and exit — rather than quietly reusing a stale database from a previous run.

New bash script

The script reads the CSV line by line and creates subdirectories like crownkill/ to save fire outputs, and cleans up after itself:

xml=$1  start_rep=$2  end_rep=$3

path="/glade/work/qasena/iLand2.0/iland-model/build/ilandc/ilandc"
output_path="/glade/derecho/scratch/qasena/output_auto/CPCRW_play"
simulation_years="5"
csv_name="/glade/work/qasena/iLand_automated/iland_reps.csv"

mkdir -p "${output_path}"

sed 1d $csv_name | while IFS=, read -r fecund gcm fri
do
    scenario_dir="${output_path}/${gcm}_${fecund}"
    mkdir -p "$scenario_dir/log"

    for rep in $(seq $start_rep $end_rep)
    do
        mkdir -p ${scenario_dir}/crownkill_${gcm}_${fecund}_${rep}
        "${path}" ${xml} $simulation_years \
            system.database.out=${gcm}_${fecund}/output_${gcm}_${fecund}_${rep}.sqlite \
            system.database.climate=${gcm}.sqlite \
            system.database.in=${fecund}.sqlite \
            system.logging.logFile=${scenario_dir}/log/log_${gcm}_${fecund}_${rep}.txt \
            modules.fire.fireReturnInterval=${fri}

        if compgen -G "${output_path}/crownkill/*" > /dev/null; then
            cp -r ${output_path}/crownkill/* \
                  ${scenario_dir}/crownkill_${gcm}_${fecund}_${rep}
            rm ${output_path}/crownkill/*
        fi
    done
done

The race condition problem

This approach worked in serial during local testing. But running replicates in parallel means multiple iLand instances share the same crownkill/ folder simultaneously.

# ❌ v1 approach — breaks with parallel instances

    system.database.out=output_${rep}.sqlite …
cp -r ${output_path}/crownkill/* \
      ${scenario_dir}/crownkill_${gcm}_${fecund}_${rep}/
rm    ${output_path}/crownkill/*

Race conditions

When multiple iLand instances run simultaneously they all write fire grids to the same crownkill/ directory. The cp of rep 1 can pick up files from rep 2, and the rm can delete files that rep 3 hasn’t copied yet.

Final solution: per-replicate XML

The fix: give each replicate its own temporary XML with its output path baked in.

This was written by AI, I think it’s correct:

iLand resolves all output paths relative to the <output> tag in the project XML. By overriding it per-replicate, the .js file’s Globals.setting('system.path.output') resolves to the correct scratch directory — with no shared state between instances.

scenario_dir="${output_path}/${gcm}_${fecund}_ep${epsilon}_dbh${dbh}_${id}/rep_${rep}"
tmp_xml="${xml_path}/${gcm}_${fecund}_ep${epsilon}_dbh${dbh}_${id}_${rep}.xml"
# 1. copy the base project XML
cp "$xml" "$tmp_xml"
# 2. rewrite the <output> tag to an absolute, replicate-specific scratch path
sed -i "s|<output>.*</output>|<output>${scenario_dir}</output>|" "$tmp_xml"
# 3. run iLand with the patched XML
"${path}" "$tmp_xml" "$simulation_years" …
# 4. clean up the temporary XML
rm "$tmp_xml"

Now

Each iLand instance writes fire grids directly to its own scenario_dir/crownkill/ — no shared directory, no race conditions.
